# Firewall / сеть: WebRTC-агент (браузер) ↔ Asterisk CC

**Сервер PBX:** `172.16.6.183` (Asterisk, Web UI, TURN)  
**Рабочие места агентов:** например `192.168.1.103` (VPN/офисная LAN)  
**Симптом:** звонок отвечается, микрофон работает, **абонента в наушниках нет**; в браузере `in=NO-inbound-rtp`, `out` растёт.

Сигналинг (WSS) уже проходит. Проблема — **UDP/RTP (голос) в сторону браузера**.

---

## 1. Рабочая станция агента (Windows)

Разрешить **входящий UDP** с сервера PBX (иначе one-way audio).

PowerShell **от администратора** (правый клик → «Запуск от имени администратора»).  
Без прав админа: `New-NetFirewallRule` → **«Отказано в доступе»** (ошибка 5).

```powershell
cd C:\Users\ADMIN\CC\asterisk-cc-phase1\scripts
.\install-agent-firewall.ps1
```

Или вручную (без ограничения LocalPort — WebRTC-порты динамические):

```powershell
New-NetFirewallRule -DisplayName "CC-Asterisk-RTP-in" -Direction Inbound -Action Allow -Protocol UDP -RemoteAddress 172.16.6.183
New-NetFirewallRule -DisplayName "CC-Asterisk-RTP-out" -Direction Outbound -Action Allow -Protocol UDP -RemoteAddress 172.16.6.183
```

Проверить:

- [ ] Правила есть в «Брандмауэр Windows» → «Правила для входящих/исходящих»
- [ ] Профили **Доменный / Частный / Публичный** — те, что реально использует VPN/LAN
- [ ] Нет блокировки Chrome/Edge отдельным AV (Kaspersky, ESET и т.д.) для **UDP**
- [ ] Временно (тест): отключить брандмауэр на 1 звонок — если звук появился, дело в локальном FW

**Исходящий TCP (уже нужен, обычно открыт):**

| Назначение | Порт | Протокол |
|------------|------|----------|
| Agent UI HTTPS | 9443 | TCP |
| SIP WebSocket (WSS) | 8089 | TCP |

## 2. Сеть между агентом и `172.16.6.183`

Маршрут с сервера до агента (пример с PBX):

```text
192.168.1.103 via 172.16.6.131 dev enp6s0f0 src 172.16.6.183
```

Проверить на **маршрутизаторе / firewall между сегментами** (в т.ч. `172.16.6.131`):

- [ ] Разрешён **UDP** `172.16.6.183` → `192.168.1.0/24` (или подсеть агентов), порты **10000–20000**, **3478**, **49160–49200**
- [ ] Разрешён **UDP** `192.168.1.0/24` → `172.16.6.183`, те же порты (исходящий голос агента)
- [ ] **Stateful / established** для UDP (ответные пакеты не режутся)
- [ ] Нет **SIP ALG** / «помощника VoIP» на промежуточном FW (ломает RTP)
- [ ] С `172.16.6.183`: `ping 192.168.1.103` (если ICMP разрешён) — не обязателен для RTP, но показывает маршрут

**Важно:** исходящий RTP с ПК агента доходит до Asterisk (~2500 пакетов в логах), **входящий RTP в браузер — нет**. Чаще всего режут **обратный UDP** на FW агента или на **172.16.6.131**.

## 3. Сервер `172.16.6.183` (хост / iptables / корп. FW)

Сервисы в **host network** (native).

| Сервис | Порт(ы) | Протокол | Направление |
|--------|---------|----------|-------------|
| Asterisk RTP | 10000–20000 | UDP | In/Out |
| TURN (coturn) | 3478 | UDP | In |
| TURN relay | 49160–49200 | UDP | In/Out |
| Asterisk WSS | 8089 | TCP | In |
| Web UI Agent | 9443 | TCP | In |
| GSM (провайдер) | 10.1.5.8/29 via enp13s4f0; 10.1.5.64/27 via enp6s0f0 | UDP SIP+RTP | см. [gsm-network-routes.md](gsm-network-routes.md) |

Проверить:

- [ ] `ss -ulnp | grep -E '3478|10000'` — слушают на `172.16.6.183`
- [ ] `systemctl is-active asterisk cc-coturn cc-webui` — все **active**
- [ ] Локальный firewall хоста не блокирует UDP между Asterisk и coturn на одном IP
- [ ] Во время звонка: в логах Asterisk есть `Got RTP packet from 192.168.1.103:<port>` (микрофон агента)

## 4. TURN (режим audio12)

В браузере ICE может показывать:

```text
local=172.16.6.183:49160  remote=172.16.6.183:12374
```

Тогда медиа идёт **через TURN на PBX**, но агенту всё равно нужен **входящий UDP с 172.16.6.183** (порты relay 49160–49200 и 3478).

- [ ] UDP **3478** от агентов к `172.16.6.183`
- [ ] UDP **49160–49200** In/Out между агентами и `172.16.6.183`

---

## 5. Критерий «исправлено»

В Chrome F12 → Console при разговоре 20+ с:

```text
[CC-RTP answered] ice=connected in=50000+ out=... local=... remote=...
```

`in` должен **расти** (не `NO-inbound-rtp`). Тогда слышен абонент.

Дополнительно: запись на сервере `recordings/.../*_russkaya_*.wav` > 300 KB и слышен абонент в файле (это уже работает).
